# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

A Conversational BI Agent: natural language ‚Üí SQL ‚Üí charts/tables, backed by DuckDB and Streamlit. Works with **any dataset** ‚Äî fully dataset-agnostic.

## Commands

```bash
# Install dependencies
pip install -r requirements.txt

# Onboard a dataset (CSV ‚Üí Parquet + AI-generated derived_tables.sql)
python setup_dataset.py

# Download Instacart from Kaggle and onboard it
python setup_dataset.py --kaggle psparks/instacart-market-basket-analysis

# Reset warehouse and reload with new data
python setup_dataset.py --reset --source /path/to/csvs

# Run the app
streamlit run app.py

# Run tests
python -m pytest tests/ -v
```

## Key Conventions

- **Model**: Always use `claude-sonnet-4-6` for all agent and setup calls
- **Environment**: Copy `.env.example` to `.env` and set `ANTHROPIC_API_KEY`
- **Dataset-agnostic**: No dataset-specific logic lives in code. All dataset-specific SQL lives in `data/derived_tables.sql` (git-ignored, generated by `setup_dataset.py`)
- **Derived tables first**: Never query raw tables directly if derived tables exist ‚Äî they are pre-aggregated for performance
- **Persistent DB**: Run `python setup_dataset.py --reset` to force full re-materialization
- **data/ is fully git-ignored**: CSVs, Parquet files, `warehouse.duckdb`, and `derived_tables.sql` are all git-ignored. The repo ships zero dataset-specific content.

## Architecture

### Data flow
```
data/*.csv  ‚Üí  Parquet  ‚Üí  DuckDB raw tables  ‚Üí  derived tables (from derived_tables.sql)  ‚Üí  agent schema  ‚Üí  LLM  ‚Üí  SQL  ‚Üí  DataFrame  ‚Üí  Plotly/st.dataframe
```

### Setup (`setup_dataset.py`)
One-time dataset onboarding script. Three phases:
1. **CSV ‚Üí Parquet conversion** ‚Äî converts any loose CSVs in `data/` to Parquet
2. **AI schema analysis** ‚Äî calls Claude with schema + sample rows ‚Üí generates `data/derived_tables.sql`
   - For Instacart (`--kaggle psparks/instacart-market-basket-analysis`): uses known-good pre-defined SQL, no AI call needed
   - For any other dataset: single Claude API call (~$0.01‚Äì0.05), one-time only
3. **Derived table build** ‚Äî optionally runs `get_connection()` to materialize tables immediately

Flags: `--source PATH`, `--kaggle SLUG`, `--reset`, `--skip-analysis`

### Database (`src/database.py`)
`get_connection()` runs two phases on startup (both are no-ops once tables exist):
1. **CSV materialization** ‚Äî `data/<name>.csv` ‚Üí `<name>.parquet` ‚Üí `<name>` DuckDB table. Automatic, filename-based.
2. **Derived table build** ‚Äî reads `data/derived_tables.sql`, parses `-- TABLE: <name>` sentinels, executes each CREATE TABLE IF NOT EXISTS in order. If the file doesn't exist, skips silently.

`_load_derived_table_sql(data_dir)` parses `derived_tables.sql` ‚Üí `list[tuple[str, str]]` (table_name, sql).

`DERIVED_TABLES` frozenset is computed at module import from `derived_tables.sql`. Empty if no file exists. Exported for agent to annotate schema.

`_schema_to_text()` annotates tables as `DERIVED ‚Äî PREFER THIS`, `RAW ‚Äî avoid`, or row count.

`validate_sql()` rejects bare `SELECT *` at query top-level. Results capped at `MAX_ROWS` (default 10,000) with a `truncated` flag.

### Agent (`src/agent.py`)
Tool-use agent: schema metadata + question ‚Üí LLM ‚Üí tool calls (run_sql) ‚Üí XML ‚Üí SQL ‚Üí execute.

**The LLM never sees data rows** ‚Äî only schema metadata and tool_result summaries for intermediate steps.

System prompt is built dynamically by `_build_system_prompt(schema)`:
- Schema text (table names, columns, row counts, DERIVED/RAW labels) ‚Äî always included
- `_build_dynamic_guidance(schema)` adds dataset-specific hints detected from the live schema:
  - No date columns detected ‚Üí warns LLM not to reference calendar fields
  - Large raw tables detected ‚Üí reminds LLM to filter aggressively

LLM output format (parsed by `_parse_response`):
```xml
<response>
  <reasoning>Step 1/2/3 planning...</reasoning>
  <chart_type>bar|line|scatter|pie|table</chart_type>
  <sql>SELECT ...</sql>
  <insight>2-3 sentence interpretation of the result.</insight>
</response>
```

On SQL execution failure, `answer_question()` makes one retry: appends the error to message history and asks the LLM to self-correct. After two failures it surfaces the error.

### Visualization (`src/visualization.py`)
Pure mapper from `chart_type` string ‚Üí Plotly figure. Chart selection is entirely the LLM's decision. Fully generic ‚Äî no hardcoded column names. Returns `None` for `"table"` or unrecognised types; `app.py` falls back to `st.dataframe`.

### UI (`app.py`)
DuckDB connection stored in `st.session_state.con` (initialized once per browser session). Chat history in `st.session_state.messages`, each entry carrying `df`, `truncated`, `chart_type`, and `insight`.

### Config (`src/config.py`)
Loads `.env` via `python-dotenv`. Key vars: `ANTHROPIC_API_KEY` (required), `DATA_DIR` (default `data/`), `DB_PATH` (default `data/warehouse.duckdb`), `MAX_ROWS` (default `10000`). `MODEL` is hardcoded to `claude-sonnet-4-6`.
