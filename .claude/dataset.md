# Dataset Guidelines

These are structural conventions every part of the system must respect, regardless of which dataset is loaded.

## Rule 1: Raw Tables May Be Too Large to Scan

For large datasets, raw CSV tables are not suitable for direct analytics queries. They may have millions of rows — scanning them will be slow and memory-intensive.

**Use derived tables for analytics when available.** They are pre-aggregated and purpose-built for the queries the agent is expected to run.

The agent's system prompt automatically annotates tables:
- `DERIVED — PREFER THIS` — pre-aggregated, fast
- `RAW — {N} rows — avoid` — large raw table, use a derived table instead
- `{N} rows` or `{N} rows (lookup table)` — small enough to query directly

## Rule 2: CSV → Table Name Mapping

`data/<name>.csv` is automatically converted to `<name>.parquet` and registered as the `<name>` table in DuckDB on startup.

Table names are derived directly from filenames. No mapping configuration is needed.

## Rule 3: Derived Tables Are Dataset-Specific

`data/derived_tables.sql` defines the pre-aggregated tables for the current dataset. This file:
- Is **git-ignored** — it belongs to the dataset, not the codebase
- Is **generated by `setup_dataset.py`** — Claude analyzes the schema and writes appropriate aggregations
- Uses a `-- TABLE: <name>` sentinel before each CREATE statement so `database.py` can parse table names

To add or modify derived tables: edit `data/derived_tables.sql` directly, then run `python setup_dataset.py --skip-analysis` (or just restart the app — tables are built on startup if they don't exist yet).

To regenerate from scratch: run `python setup_dataset.py --reset`.

## Rule 4: No Assumptions About Column Names

The agent system prompt is built dynamically from the live schema — it lists whatever columns actually exist. Do not add code that assumes specific column names (like `product_id`, `order_id`, `days_since_prior_order`) unless those columns are verified to exist in the current dataset.

The agent uses `run_sql` to explore unfamiliar schemas before writing final SQL.

## Rule 5: Date/Time Detection Is Automatic

`_build_dynamic_guidance()` in `src/agent.py` inspects column names at runtime:
- If no date/timestamp columns are detected → the system prompt warns the LLM not to reference calendar fields
- If date columns exist → no warning, LLM can use date filters normally

This means the agent adapts automatically to datasets with or without calendar dates.

## Rule 6: Switching Datasets

To replace the current dataset with a new one:

```bash
python setup_dataset.py --reset --source /path/to/new/csvs
# or for Kaggle:
python setup_dataset.py --reset --kaggle <slug>
```

`--reset` removes `warehouse.duckdb`, all `.parquet` files, and `derived_tables.sql`. The new dataset is loaded from scratch and a new `derived_tables.sql` is generated.
